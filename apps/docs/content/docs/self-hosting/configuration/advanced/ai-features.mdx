## Overview

This guide covers how to enable the AI recipient and field detection features when you self-host Documenso.

## What This Enables

- Detect recipients from uploaded PDFs (roles, names, emails when present).
- Detect and place fields (signature, initials, name, email, date, text, number, radio, checkbox) onto draft envelopes.
- Built-in rate limits (3 requests per minute per IP) to prevent abuse.

## Prerequisites

- One configured AI provider:
  - Google Vertex AI with the **Vertex AI API** enabled and billing active.
  - OpenAI API access with a vision-capable model.
  - Ollama running a local vision-capable model.
- Documenso version that includes the AI detection feature and the corresponding database migration.

## Configure Environment Variables

Add one of these provider configurations to your deployment `.env` (or secret manager):

### Google Vertex AI

```
AI_PROVIDER="google"
GOOGLE_VERTEX_PROJECT_ID="<your-gcp-project-id>"
GOOGLE_VERTEX_API_KEY="<your-vertex-api-key>"
# Optional, defaults to "global"
GOOGLE_VERTEX_LOCATION="global"
# Optional, defaults to "gemini-3-flash-preview"
GOOGLE_VERTEX_MODEL="gemini-3-flash-preview"
```

### OpenAI

```
AI_PROVIDER="openai"
OPENAI_API_KEY="<your-openai-api-key>"
# Optional, defaults to "gpt-4.1-mini"
OPENAI_MODEL="gpt-4.1-mini"
# Optional, for OpenAI-compatible proxies or hosted providers
OPENAI_BASE_URL=""
OPENAI_ORGANIZATION=""
OPENAI_PROJECT=""
```

### Ollama

```
AI_PROVIDER="ollama"
OLLAMA_BASE_URL="http://localhost:11434/v1"
OLLAMA_MODEL="llama3.2-vision"
# Ollama does not require a real API key, but OpenAI-compatible clients require a value.
OLLAMA_API_KEY="ollama"
```

<Callout type="info">
  Detection sends rendered PDF pages as images, so OpenAI and Ollama models must support vision/image inputs.
  For Docker deployments, `localhost` inside the Documenso container points to the container itself. Use
  `http://host.docker.internal:11434/v1` or the Ollama service name, such as `http://ollama:11434/v1`.
</Callout>

## Deploy with the Published Container

- Use the official Documenso image (DockerHub or GHCR) and supply the provider env vars above.
- Ensure migrations run on startup (the container runs `prisma migrate deploy` in production mode).
- Restart the container after adding or changing provider env vars.

## Enable the Feature in Documenso

Once the service is running with the AI provider env vars:

{/* prettier-ignore */}
<Steps>
<Step>
### Organisation settings

Go to **Settings** → **Document Preferences** → **AI Features** and set to **Enabled**.

Teams that inherit organisation defaults will get this automatically.

</Step>
<Step>
### Team settings

If a team overrides organisation defaults:

- Go to **Team Settings** → **Document Preferences** → **AI Features**
- Choose **Enabled**, or **Inherit** to follow the organisation

</Step>
<Step>
### Verify in the editor

Open a draft envelope and check:

- **Recipients**: sparkle button for AI detection is visible
- **Fields**: **Detect with AI** is available

</Step>
</Steps>

## Troubleshooting

<Accordions type="multiple">
  <Accordion title="Too many requests">
    Wait a minute or two and retry (rate limit is 3/min per IP).
  </Accordion>
  <Accordion title="AI options hidden">
    Ensure the env vars are set, the server was restarted after setting them, and
    `aiFeaturesEnabled` is enabled at organisation/team level.
  </Accordion>
  <Accordion title="Detection fails immediately">
    Confirm the provider key or local model is valid. For Ollama, confirm the model is pulled and
    `OLLAMA_BASE_URL` is reachable from the Documenso container. Check server logs for provider status codes.
  </Accordion>
</Accordions>

If issues persist, recheck env vars, restart the service, and confirm the Prisma migration was applied.
